# 统一运维管理平台一期 Spec

## Why
企业 IT 资产分散、监控孤岛、告警风暴等问题导致运维效率低下。需要构建统一运维管理平台，一期聚焦 CMDB 配置管理、监控系统、告警平台三大核心模块，实现资产-监控-告警的闭环管理。

## What Changes
- 新建 CMDB 配置管理模块：支持自定义模型组、手工录入/API 采集资产、资产分组与生命周期管理
- 新建监控系统模块：主机/网络/中间件指标采集、时序数据存储、可视化仪表盘
- 新建告警平台模块：告警规则引擎、告警收敛/抑制/升级、多渠道通知、告警处理闭环
- 新建联系人管理模块：管理通知联系人、通知渠道（短信/邮件/电话/企业微信/钉钉）
- 新建工单流程管理模块：告警触发后自动生成运维工单，支持状态流转、处理记录、闭环关闭告警
- 新建公共基础模块：用户认证、RBAC 权限、审计日志、系统配置

## Delivery Status
- M0-M2 基本功能已实现：项目骨架、公共基础模块、CMDB 模型/资产/API 同步已具备本地验收基础。
- M3-M7 尚未开始开发：监控、告警、联系人通知、工单流程、验证验收和部署交接仍处于设计与计划阶段。

## Impact
- 这是一个全新项目，无已有代码受影响
- 技术栈：Go 后端 + Vue/React 前端 + PostgreSQL + VictoriaMetrics + Redis + NATS

## ADDED Requirements

### Requirement: CMDB 配置管理数据库
系统 SHALL 提供自定义模型组能力，支持用户创建模型组、定义模型字段（文本/数字/枚举/日期/IP/关联引用），预置常用模型（物理服务器/虚拟机/交换机/路由器/Nginx/MySQL/Redis/业务应用）。

系统 SHALL 支持手工录入（表单+Excel/CSV 批量导入）和通用 API 采集（增量/全量同步）两种资产录入方式。

系统 SHALL 支持资产分组（按业务线/环境/地域）、资产生命周期状态管理（采购→上架→运行→维修→报废）、资产变更历史记录。

#### Scenario: 创建自定义模型组
- **WHEN** 运维管理员创建名为"计算资源"的模型组，并在其下定义"物理服务器"模型
- **THEN** 系统保存模型组和模型定义，前端根据字段定义动态渲染资产录入表单

#### Scenario: 通过 API 同步资产
- **WHEN** 外部系统调用 /api/v1/cmdb/assets/sync 接口推送资产数据
- **THEN** 系统验证 API Key 认证，按 unique_key 匹配并增量更新资产属性

#### Scenario: 资产详情页展示关联信息
- **WHEN** 用户查看某台服务器的资产详情
- **THEN** 系统展示该资产的基本信息、关联的实时监控面板、当前活跃告警和历史告警记录

### Requirement: 监控系统
系统 SHALL 采集主机（CPU/内存/磁盘/网络）、网络设备（SNMP/IPMI）、中间件（Nginx/MySQL/Redis/Kafka）的运行指标，存入时序数据库（VictoriaMetrics）。

系统 SHALL 提供预置仪表盘（Linux 主机概览/网络设备概览/MySQL/Redis/Nginx 面板）和自定义仪表盘（拖拽布局/变量替换/时间范围选择/分享）。

系统 SHALL 支持类 PromQL 查询语法，支持按 CMDB 资产筛选指标。监控数据通过 asset_id 与 CMDB 资产关联。

#### Scenario: 主机 Agent 采集指标
- **WHEN** 主机 Agent 部署完成后
- **THEN** Agent 自动采集 CPU/内存/磁盘/网络指标并上报至 Collector，资源占用 CPU<1%、内存<50MB

#### Scenario: 查看自定义仪表盘
- **WHEN** 运维工程师创建包含 CPU 使用率折线图的仪表盘并选择特定主机
- **THEN** 系统展示该主机对应时间范围的 CPU 使用率趋势图

---

### Requirement: 告警平台 - 规则引擎
系统 SHALL 支持基于监控指标创建告警规则，规则包含：指标名、比较运算符、阈值、持续时间、聚合方式。告警等级分为 P0(紧急)、P1(重要)、P2(一般)、P3(提醒)。

系统 SHALL 支持告警规则按 CMDB 资产分组生效，支持规则启停和静默（维护窗口）。

#### Scenario: 创建告警规则
- **WHEN** 运维管理员创建规则"CPU使用率 > 90% 持续5分钟，等级 P1，作用于生产环境资产组"
- **THEN** 系统保存规则并开始在每轮评估周期中检查该规则

---

### Requirement: 告警平台 - 通知分发
系统 SHALL 根据告警等级选择通知方式：
- P0（紧急）：电话 + 短信 + 邮件
- P1（重要）：短信 + 邮件
- P2（一般）：邮件
- P3（提醒）：邮件（可选）

通知对象为告警设备在 CMDB 中关联的负责人。

系统 SHALL 提供联系人管理模块，支持管理通知联系人及其通知渠道配置（短信、邮件、电话、企业微信、钉钉），每个联系人可配置多个通知渠道。

#### Scenario: P0 告警触发通知
- **WHEN** 核心数据库触发 P0 级宕机告警
- **THEN** 系统查询该设备在 CMDB 中的负责人，通过电话+短信+邮件三种渠道同时发送告警通知

#### Scenario: P1 告警触发通知
- **WHEN** 生产环境某服务器触发 P1 级 CPU 告警
- **THEN** 系统查询该服务器在 CMDB 中的负责人，通过该负责人配置的短信+邮件渠道发送告警通知

#### Scenario: P2 告警触发通知
- **WHEN** 测试环境某服务器触发 P2 级磁盘使用率告警
- **THEN** 系统查询该服务器在 CMDB 中的负责人，通过该负责人配置的邮件渠道发送告警通知

#### Scenario: 管理联系人通知渠道
- **WHEN** 管理员为联系人"张三"配置通知渠道：短信(138xxxx)、邮件(zhangsan@company.com)、企业微信
- **THEN** 系统保存联系人及其多渠道配置，后续告警通知按渠道配置发送

### Requirement: 工单流程管理 - 告警处理闭环
系统 SHALL 支持完整的告警到工单处理闭环流程：

1. **告警触发**：监控指标异常，规则引擎匹配触发告警事件
2. **负责人识别**：系统根据告警关联的设备，从 CMDB 中查找该设备的负责人
3. **通知发送**：根据告警等级，通过负责人配置的通知渠道（电话/短信/邮件/企业微信/钉钉）发送告警通知
4. **工单生成**：告警触发或通知发送后，系统自动创建运维工单，关联告警信息和资产信息
5. **工单跟踪**：运维人员处理工单，记录处理过程和结果，工单状态流转：待处理 → 处理中 → 已完成 → 已闭环
6. **工单闭环**：运维人员确认问题已解决，工单标记为"已闭环"，关闭后不可继续追加处理记录
7. **告警关闭**：工单闭环后，系统自动将关联的告警状态更新为 Closed

工单 SHALL 包含：关联告警信息、告警等级、关联资产信息（含负责人）、处理人、处理记录、处理结果、闭环时间。

系统 SHALL 提供工单列表和工单详情，支持按状态、告警等级、关联资产、负责人、创建时间筛选，并展示处理记录时间线。

#### Scenario: 告警触发完整处理流程
- **WHEN** 生产环境服务器触发 P1 级 CPU 告警
- **THEN** 系统依次执行：①从 CMDB 查找该服务器负责人 → ②通过短信+邮件通知负责人 → ③自动创建运维工单（状态：待处理）

#### Scenario: 工单处理与闭环
- **WHEN** 运维人员在工单中记录处理过程，确认问题已解决并标记工单为"已闭环"
- **THEN** 系统将关联的告警状态自动更新为 Closed，记录闭环时间

#### Scenario: 工单跟踪
- **WHEN** 运维经理查看待处理工单列表
- **THEN** 系统展示所有未闭环工单，包含告警等级、关联资产、负责人、处理时长、当前状态

---

### Requirement: 联系人管理模块
系统 SHALL 提供独立的联系人管理功能，支持联系人的增删改查。每个联系人关联一个系统用户，可配置多个通知渠道（短信、邮件、电话、企业微信 Webhook、钉钉 Webhook）。

#### Scenario: 创建联系人
- **WHEN** 管理员创建联系人并选择关联用户"李四"，配置邮件和短信渠道
- **THEN** 系统保存联系人信息，后续告警通知可通过邮件和短信发送给李四

---

### Requirement: 公共基础模块
系统 SHALL 提供用户认证（用户名/密码，预留 LDAP/AD 接口）、RBAC 权限（系统管理员/运维管理员/运维操作员/只读用户）、操作审计日志、系统配置管理。

#### Scenario: 用户登录
- **WHEN** 用户输入正确的用户名和密码
- **THEN** 系统验证通过，返回 JWT Token，记录登录审计日志
