# WingOps 项目约束规范

## 文档信息

| 项目 | 内容 |
|------|------|
| 适用范围 | WingOps 后端服务及其配套接口开发 |
| 当前后端框架 | Go + Gin |
| 目标 | 保证项目能启动、接口能对接、业务能迭代、问题能排查 |

## 1. 基本原则

- 先满足能运行、能配置、能验证，再扩展业务能力。
- 新增功能优先复用当前项目结构和框架机制，不为了单个功能新造一套目录、响应格式或错误码体系。
- 每个接口变更都要能回答 4 个问题：路由在哪里？请求如何校验？业务规则在哪里？数据如何读写？
- 文档、配置样板和启动说明必须随代码一起更新，不能只在本地环境里可用。

## 2. 启动线：后端必须能正常跑起来

这条线解决的是：这个后端能不能在新环境里正常启动，并且能确认基础依赖是否可用。

### 2.1 配置来源

当前项目通过 `internal/config/config.go` 使用 Viper 读取配置：

- 默认读取 `./configs/app.yaml` 或当前目录下的 `app.yaml`。
- 配置样板放在 `configs/app.example.yaml`。
- 环境变量前缀为 `WINGOPS`，配置键中的 `.` 会映射为 `_`。

示例：

| 配置项 | YAML 键 | 环境变量 |
|--------|---------|----------|
| 服务监听地址 | `server.addr` | `WINGOPS_SERVER_ADDR` |
| PostgreSQL 地址 | `postgres.dsn` | `WINGOPS_POSTGRES_DSN` |
| Redis 地址 | `redis.addr` | `WINGOPS_REDIS_ADDR` |
| NATS 地址 | `nats.url` | `WINGOPS_NATS_URL` |
| JWT 密钥 | `jwt.secret` | `WINGOPS_JWT_SECRET` |
| Token 有效期 | `jwt.access_token_ttl_minutes` | `WINGOPS_JWT_ACCESS_TOKEN_TTL_MINUTES` |

### 2.2 禁止写死运行配置

端口、数据库地址、Redis 地址、NATS 地址、JWT 密钥等运行环境相关配置不得写死在业务代码里。

允许存在的默认值只用于本地开发兜底，生产和测试环境必须通过配置文件或环境变量覆盖。新增配置项时必须同步更新：

- `internal/config/config.go` 中的配置结构、默认值和读取逻辑。
- `configs/app.example.yaml` 中的配置样板。
- `README.md` 中的启动和配置说明。

### 2.3 启动入口和依赖检查

当前后端启动入口是 `cmd/server/main.go`。启动时必须至少完成：

- 读取配置。
- 初始化日志。
- 连接 PostgreSQL。
- 执行数据库迁移。
- 初始化 Repository、Service、Handler 和 Router。
- 按配置监听端口。

数据库连接失败、迁移失败、种子数据初始化失败时，服务必须输出日志并退出，不能静默降级成不可预期状态。

### 2.4 健康检查

当前健康检查接口为：

```text
GET /healthz
```

返回示例：

```json
{
  "status": "ok"
}
```

新增部署、网关或监控配置时，默认使用 `/healthz` 作为存活检查。如果确实需要 `/health`，应作为兼容别名新增，不能移除已有 `/healthz`。

### 2.5 启动线验收

新增或修改启动相关代码后，至少验证：

```bash
make run
curl http://localhost:8080/healthz
```

涉及数据库、缓存或消息队列时，先启动依赖：

```bash
make infra-up
```

## 3. 接口线：接口规则必须统一

这条线解决的是：以后前端接口测试和继续加功能时，能不能按同一套规则对接。

### 3.1 路由放置位置

当前项目的路由注册入口在 `internal/http/router.go`：

- API 基础路径统一使用 `/api/v1`。
- 健康检查放在根路由 `/healthz`，不挂在 `/api/v1` 下。
- 业务模块在各自 Handler 中提供 `Register*Routes` 方法。
- 认证和权限中间件在 `internal/http/middleware`。

新增接口时，不要在 `cmd/server/main.go` 里直接注册业务路由。应在对应模块 Handler 中注册路由，再由 `internal/http/router.go` 统一挂载。

### 3.2 成功响应格式

当前代码已普遍使用：

```json
{
  "code": 0,
  "data": {}
}
```

新增接口应继续保持 `code = 0` 表示成功，并明确区分两类成功场景。

#### 列表成功

列表接口必须返回数组。如果涉及分页，必须同时返回分页信息和总数。

无分页列表：

```json
{
  "code": 0,
  "data": [
    {
      "id": "1",
      "name": "example"
    }
  ]
}
```

分页列表：

```json
{
  "code": 0,
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 20
  }
}
```

约束：

- `items` 永远是数组，空列表返回 `[]`，不能返回 `null`。
- `total` 表示符合筛选条件的总数，不是当前页数量。
- `page` 和 `page_size` 使用请求实际生效的值。
- 是否分页由业务需要决定，但同一资源的列表接口不要混用多套字段名。

#### 对象成功

详情、创建后返回数据、更新后返回数据都属于对象成功。

```json
{
  "code": 0,
  "data": {
    "id": "1",
    "name": "example"
  }
}
```

约束：

- 创建接口优先返回创建后的完整对象。
- 更新接口优先返回更新后的完整对象。
- 删除接口当前可返回 `{ "deleted": true }`，后续若统一为无内容响应，需要在接口文档中明确并保持兼容。

### 3.3 失败响应格式

失败响应使用：

```json
{
  "code": 400,
  "message": "invalid request"
}
```

`message` 面向前端和排查使用，应简洁说明失败原因。不要把数据库错误、堆栈、密钥、连接串等内部信息直接返回给客户端。

### 3.4 HTTP 状态码

状态码优先复用框架规范和通用 HTTP 标准。

| 状态码 | 场景 |
--------|------|
| 200 | 查询、更新、删除成功；当前项目创建成功也可继续使用 200 |
| 201 | 新增资源成功；如后续切换为 201，同一模块内要保持一致 |
| 400 | 参数错误、请求体格式错误、业务输入不合法 |
| 401 | 未登录、Token 缺失、Token 无效 |
| 403 | 已登录但无权限 |
| 404 | 资源不存在 |
| 409 | 唯一约束冲突、状态冲突、重复操作 |
| 500 | 未预期的服务端异常 |

接口层不要为了所有错误都返回 200。HTTP 状态码表示请求处理结果的大类，业务错误码只用于补充细节。

### 3.5 业务错误码规则

当前项目主要使用 HTTP 状态码作为 `code`。如果后续确实需要业务错误码，必须先定义规则，再落地代码。

建议规则：

- `code = 0` 固定表示成功。
- `code` 为 HTTP 状态码时，只表达通用错误，例如 `400`、`401`、`403`、`404`、`500`。
- 业务错误码必须按模块分段，不能随手新增。
- 同一错误在不同接口中必须使用同一个 `code` 和稳定的 `message`。

示例分段：

| 范围 | 模块 |
|------|------|
| 10000-10999 | 认证与用户 |
| 11000-11999 | 权限与角色 |
| 20000-20999 | CMDB 模型 |
| 21000-21999 | CMDB 资产 |
| 30000-30999 | 系统配置与审计 |

落地业务错误码前，需要先补充错误码表文档，不能在 Handler 里想到哪写到哪。

### 3.6 接口线验收

新增接口至少检查：

- 路由是否挂在 `internal/http/router.go` 的 `/api/v1` 分组下。
- 成功响应是否区分列表和对象。
- 参数错误、未登录、无权限、资源不存在、系统异常是否使用一致的响应格式。
- 前端 API 调用和接口文档是否同步更新。

## 4. 业务线：请求、业务规则、数据库访问不能堆在一起

这条线解决的是：以后新增功能时，不会把请求解析、业务判断和数据库操作全堆在一个函数里。

### 4.1 当前项目分层

当前后端是 Go + Gin，主要分层如下：

| 职责 | 当前项目位置 | 说明 |
|------|--------------|------|
| 请求入口和响应输出 | `internal/*/*_handler.go` | 例如 `internal/auth/handler.go`、`internal/cmdb/asset_handler.go` |
| 路由组装 | `internal/http/router.go` | 统一挂载 `/api/v1`、认证、权限和审计中间件 |
| 业务规则 | `internal/*/service.go` 或模块内业务文件 | 例如认证模块使用 `internal/auth/service.go` |
| 数据访问接口 | `internal/*/repository.go` | 定义 Repository 接口 |
| PostgreSQL 实现 | `internal/*/postgres_repository.go` | 负责 SQL 和数据库读写 |
| 模型定义 | `internal/*/model.go` | 定义领域对象、请求外的业务数据结构 |
| 中间件 | `internal/http/middleware` | 认证、权限、审计等横切逻辑 |

不同框架可能叫 Controller、Handler、Service、Repository、DAO。本项目按 Go + Gin 的当前习惯解释为 Handler、Service、Repository。新增功能时应沿用这些概念；如果某个轻量模块没有 Service，也必须说明业务规则放在哪里，不能把规则散落到多个 Handler 里。

### 4.2 Handler 只处理请求和响应

Handler 可以做：

- 读取路径参数、Query 参数、Header 和 JSON 请求体。
- 调用 Gin 的参数绑定和基础校验。
- 调用 Service 或 Repository。
- 把结果转换为统一响应。
- 把错误映射为统一 HTTP 状态码和错误响应。

Handler 不应做：

- 拼 SQL 或直接访问数据库连接。
- 承载复杂业务流程、状态流转规则或跨表判断。
- 生成难以复用的权限规则。
- 把多个模块的数据读写直接串在一个请求函数里。

### 4.3 业务规则放到业务层

新增复杂业务时，应优先增加或复用 Service：

- 认证、登录、用户、角色等规则放在 `internal/auth/service.go`。
- 后续监控、告警、工单等模块如有状态流转、规则判断、跨 Repository 协作，应建立对应 Service。
- CMDB 当前部分 Handler 直接调用 Repository。后续如果新增逻辑超过简单 CRUD，应补 Service，不继续把规则堆进 Handler。

业务层负责判断「能不能做」和「怎么做」，数据层只负责「怎么存取」。

### 4.4 数据库访问放到数据层

数据库读写必须通过 Repository：

- Repository 接口放在 `internal/<module>/repository.go`。
- PostgreSQL 实现放在 `internal/<module>/postgres_repository.go`。
- SQL、事务、扫描行、处理数据库约束错误等逻辑只出现在 Repository 实现中。
- Handler 和 Service 不直接依赖具体 SQL。

如果新增模块暂时使用内存实现，也应保持 Repository 接口，方便测试和后续切换 PostgreSQL。

### 4.5 参数校验和权限校验

参数校验分两层：

- 请求格式、必填字段、基础类型校验放在 Handler，使用 Gin binding 或明确的解析逻辑。
- 业务合法性校验放在 Service，例如唯一性、状态是否允许变更、关联资源是否存在。

权限校验优先使用当前中间件：

- 登录校验：`middleware.Auth`。
- 权限码校验：`middleware.RequirePermission("module.resource.action")`。
- 权限码命名继续使用当前格式，例如 `cmdb.asset.read`、`cmdb.asset.write`。

不要在每个 Handler 里手写一套权限判断。确实需要资源级权限时，应先说明框架机制如何承载，例如新增中间件、Service 规则或 Repository 查询约束。

### 4.6 业务线验收

新增功能至少检查：

- Handler 是否只做请求、响应和错误映射。
- 业务规则是否集中在 Service 或模块明确的业务文件。
- 数据库读写是否通过 Repository。
- 权限是否通过当前中间件或清晰的业务层规则承载。
- 测试是否覆盖关键业务规则，而不是只测路由是否能返回 200。

## 5. 运维线：日志、错误处理、依赖入口必须清楚

这条线解决的是：项目出问题时，能不能启动，能不能查日志，能不能知道数据库和权限这些基础入口在哪里。

### 5.1 日志

当前项目在 `cmd/server/main.go` 使用 Zap 初始化生产日志。

要求：

- 服务启动、监听地址、数据库连接失败、迁移失败、服务退出必须有日志。
- 服务端异常记录到日志，客户端响应只返回必要错误信息。
- 不在日志中输出明文密钥、Token、数据库密码、完整连接串等敏感信息。
- 后续如增加请求日志，应通过 Gin 中间件统一处理，不在每个 Handler 里重复打印。

### 5.2 统一错误处理

当前项目错误响应由 Handler 和中间件直接写出。新增代码应至少保持同一响应结构：

```json
{
  "code": 400,
  "message": "invalid request"
}
```

后续如果抽取统一错误处理函数，应放在公共包或模块内复用函数中，并保证：

- 参数错误统一为 400。
- 未登录统一为 401。
- 无权限统一为 403。
- 资源不存在统一为 404。
- 未预期异常统一为 500，并记录服务端日志。

不要让同一类错误在不同 Handler 中返回不同字段名。

### 5.3 数据库连接和迁移

数据库入口在：

- 连接：`internal/database/postgres.go`。
- 迁移：`migrations/*.sql`。
- 启动调用：`cmd/server/main.go`。

新增数据库表或字段时必须：

- 新增迁移文件。
- 更新相关 Repository。
- 必要时更新种子数据。
- 运行 `make test` 或至少运行相关模块测试。

数据库连接必须在服务启动时可验证。连接失败不能等到第一个业务请求才暴露。

### 5.4 权限入口

权限入口在：

- 中间件：`internal/http/middleware/auth.go`。
- 路由挂载：`internal/http/router.go`。
- 权限数据：认证和 RBAC 相关迁移、Repository、Service。

新增受保护接口时必须明确权限码。读写权限应拆开，例如：

- `module.resource.read`
- `module.resource.write`

暂时不做细粒度权限时，也要保留清晰占位，不能直接绕过认证和权限中间件。

### 5.5 README 必须覆盖基础运维信息

`README.md` 至少要说明：

- 如何启动依赖服务。
- 如何配置后端。
- 如何启动后端。
- 如何启动前端。
- 如何运行测试。
- 健康检查地址。
- 本地默认账号或开发凭据。

新增基础设施依赖、配置项、启动命令或测试命令时，必须同步更新 README。

### 5.6 运维线验收

每次新增后端能力后，至少确认：

```bash
make test
make run
curl http://localhost:8080/healthz
```

如果改动涉及数据库：

- 确认迁移能从空库执行。
- 确认 Repository 测试或相关接口测试覆盖核心读写路径。

如果改动涉及权限：

- 确认未登录返回 401。
- 确认无权限返回 403。
- 确认有权限可以访问。

## 6. 新增后端功能检查清单

提交新增后端功能前，逐项确认：

- 配置项没有写死，样板和 README 已更新。
- 服务能启动，`/healthz` 可访问。
- 路由挂载位置符合 Gin 当前结构。
- 成功响应区分列表和对象。
- 失败响应字段和 HTTP 状态码统一。
- Handler 没有堆业务规则和 SQL。
- 业务规则有明确承载位置。
- 数据访问走 Repository。
- 受保护接口有权限码。
- 日志和错误处理能支持排查问题。
- 测试、启动或手工验证步骤已执行并记录。
