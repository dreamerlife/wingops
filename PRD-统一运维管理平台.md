## 1. 产品概述

### 1.1 背景

随着企业 IT 基础设施规模不断扩大，服务器、网络设备、中间件等资产数量持续增长，运维团队面临以下核心痛点：

- **资产分散**：资产配置信息分散在不同系统和表格中，缺乏统一管理视图，资产数据不一致、不完整
- **监控孤岛**：各类监控工具独立运行，监控数据无法与资产关联，故障定位效率低
- **告警风暴**：告警规则分散，缺乏统一管理和智能降噪，关键告警被淹没
- **协作低效**：运维操作依赖口头沟通和手工操作，缺乏标准化流程和自动化能力

### 1.2 产品目标

构建一套统一运维管理平台，实现：

1. **资产统一管理**：建立企业级 CMDB，统一管理所有 IT 资产，支持自定义模型和多种数据采集方式
2. **全栈可观测**：覆盖主机、网络、中间件的全栈监控，监控数据与 CMDB 资产自动关联
3. **智能告警**：统一告警规则管理，支持告警收敛、降噪、升级和通知，减少告警疲劳
4. **数据驱动运维**：以 CMDB 为核心，打通监控、告警数据，实现从资产视角的运维全景视图

### 1.3 价值主张

| 价值维度 | 描述 |
|---------|------|
| 效率提升 | 统一平台减少工具切换，资产-监控-告警联动缩短故障定位时间 |
| 数据准确 | CMDB 作为唯一数据源，消除多系统数据不一致问题 |
| 降低风险 | 及时告警和告警升级机制，避免关键故障被遗漏 |
| 可扩展 | 自定义模型组设计，灵活适配不同企业的资产管理需求 |

### 1.4 产品定位

- **目标用户**：中大型企业（500-5000+ 资产规模，10-50 人运维团队）
- **一期范围**：CMDB 配置管理 + 监控系统 + 告警平台 + 工单流程管理
- **后续规划**：高级流程引擎、运维自动化（详见第 8 章）

## 2. 用户与角色分析

### 2.1 目标用户画像

| 角色 | 职责描述 | 核心诉求 | 使用频率 |
|------|---------|---------|---------|
| **运维工程师** | 日常运维操作、故障处理、巡检 | 快速发现故障、高效定位问题、减少重复操作 | 每日高频 |
| **高级运维工程师** | 架构优化、监控策略制定、自动化建设 | 灵活配置监控规则、自定义仪表盘、数据分析 | 每日中频 |
| **运维经理** | 团队管理、运维流程制定、KPI 考核 | 运维全景视图、SLA 报表、团队工作量统计 | 每周中频 |
| **IT 管理员** | 资产管理、权限管理、系统配置 | 资产全生命周期管理、权限精细化控制 | 每周低频 |
| **业务负责人** | 业务系统可用性关注 | 业务视角的可用性看板、重大故障通知 | 按需查看 |

### 2.2 权限模型

采用 **RBAC（基于角色的访问控制）** 模型：

- **系统管理员**：平台全部权限，包括用户管理、角色配置、系统设置
- **运维管理员**：CMDB 模型管理、监控策略配置、告警规则管理
- **运维操作员**：资产查看/编辑、监控查看、告警处理
- **只读用户**：仅查看权限，适用于业务负责人等角色

支持**资源级权限**控制：可按资产分组、业务线等维度控制数据的可见范围。

### 3.3 模块关系

- **CMDB** 是核心基础，为监控和告警提供资产元数据和拓扑关系
- **监控系统** 从采集层收集指标数据，与 CMDB 资产关联存储
- **告警平台** 基于监控数据执行规则判断，告警事件关联 CMDB 资产信息
- **工单流程管理** 承接告警处理，记录处理过程、结果和闭环状态
- 四者形成 **资产 → 监控 → 告警 → 工单闭环** 的数据链路

## 4. 一期功能模块详细设计

### 4.1 CMDB 配置管理数据库

#### 4.1.1 功能概述

CMDB 是平台的核心基础模块，提供 IT 资产的统一管理能力。支持自定义配置模型组，适配不同企业的资产管理需求。资产数据可通过手工录入和通用 API 两种方式采集。

#### 4.1.2 功能清单

| 功能模块 | 功能项 | 优先级 | 描述 |
|---------|--------|--------|------|
| 模型管理 | 模型组定义 | P0 | 支持创建自定义模型组（如：服务器、网络设备、中间件、业务应用等） |
| 模型管理 | 模型定义 | P0 | 在模型组下定义具体模型，包含字段定义（名称、类型、是否必填、默认值等） |
| 模型管理 | 字段类型 | P0 | 支持文本、数字、枚举、日期、IP地址、关联引用等字段类型 |
| 模型管理 | 模型关系 | P1 | 定义模型间的关联关系（如：服务器 → 运行 → 中间件） |
| 模型管理 | 预置模型 | P0 | 系统预置常用模型组：物理服务器、虚拟机、网络设备、中间件、数据库、业务系统 |
| 资产管理 | 资产 CRUD | P0 | 基于模型定义的资产增删改查操作 |
| 资产管理 | 手工录入 | P0 | 通过表单界面手工录入资产信息 |
| 资产管理 | 批量导入 | P1 | 支持 Excel/CSV 文件批量导入资产 |
| 资产管理 | API 采集 | P0 | 提供标准 REST API 接口，支持外部系统推送资产数据 |
| 资产管理 | 资产变更历史 | P1 | 记录资产属性的每次变更，支持变更审计和回溯 |
| 资产管理 | 资产生命周期 | P1 | 支持资产状态管理：采购 → 上架 → 运行 → 维修 → 报废 |
| 资产管理 | 资产分组 | P0 | 支持按业务线、环境（生产/测试/开发）、地域等维度分组 |
| 关系管理 | 拓扑视图 | P2 | 可视化展示资产间的关联关系拓扑图 |
| 关系管理 | 影响分析 | P2 | 基于关系拓扑，分析某资产故障的影响范围 |
| 数据发现 | 自动发现 | P2 | 通过网络扫描自动发现新资产（二期考虑） |

#### 4.1.3 自定义模型组设计

**核心概念：**

```
模型组 (Model Group)
 └── 模型 (Model)
      ├── 字段定义 (Field Definition)
      │    ├── 字段名、显示名、类型
      │    ├── 是否必填、是否唯一
      │    ├── 默认值、枚举值列表
      │    └── 校验规则（正则、范围等）
      └── 关系定义 (Relation)
           ├── 关联目标模型
           └── 关系类型（运行于、依赖、包含等）
```

**预置模型组示例：**

| 模型组 | 模型 | 核心字段 |
|--------|------|---------|
| 计算资源 | 物理服务器 | 主机名、管理IP、SN序列号、CPU型号、内存容量、磁盘容量、操作系统、机房、机柜位置 |
| 计算资源 | 虚拟机 | 主机名、管理IP、vCPU数、内存大小、磁盘大小、操作系统、宿主机、所属集群 |
| 网络设备 | 交换机 | 设备名、管理IP、设备型号、端口数、固件版本、机房、所属区域 |
| 网络设备 | 路由器 | 设备名、管理IP、设备型号、接口数、固件版本、机房 |
| 中间件 | Nginx | 实例名、版本、监听端口、配置文件路径、所属服务器 |
| 中间件 | MySQL | 实例名、版本、端口、数据目录、角色（主/从）、所属服务器 |
| 中间件 | Redis | 实例名、版本、端口、角色（主/从/sentinel）、所属服务器 |
| 业务系统 | 业务应用 | 应用名、负责人、团队、SLA等级、关联服务器列表、关联中间件列表 |

**自定义模型 API 设计：**

```
POST   /api/v1/cmdb/model-groups              # 创建模型组
GET    /api/v1/cmdb/model-groups              # 查询模型组列表
PUT    /api/v1/cmdb/model-groups/{id}          # 更新模型组
DELETE /api/v1/cmdb/model-groups/{id}          # 删除模型组

POST   /api/v1/cmdb/model-groups/{gid}/models  # 创建模型
GET    /api/v1/cmdb/models/{id}                # 查询模型详情（含字段定义）
PUT    /api/v1/cmdb/models/{id}                # 更新模型
DELETE /api/v1/cmdb/models/{id}                # 删除模型
```

#### 4.1.4 资产录入方式

**方式一：手工录入**
- 根据模型定义动态渲染表单
- 支持单条录入和批量 Excel/CSV 导入
- 导入时进行字段校验，返回校验错误明细

**方式二：API 采集**
- 提供标准 REST API，支持外部系统（如云平台、虚拟化平台）主动推送资产数据
- 支持增量更新和全量同步两种模式
- API 认证方式：API Key + Secret 签名
- 请求示例：

```json
POST /api/v1/cmdb/assets/sync
{
  "model_id": "physical_server",
  "sync_mode": "incremental",
  "assets": [
    {
      "unique_key": "sn:ABC123456",
      "attributes": {
        "hostname": "web-server-01",
        "management_ip": "10.0.1.101",
        "os": "CentOS 7.9",
        "cpu_cores": 64,
        "memory_gb": 256
      }
    }
  ]
}
```

#### 4.1.5 与监控关联

- 每个 CI（配置项）可关联一个或多个监控对象
- 关联方式：通过唯一标识符（如主机名、管理 IP）自动匹配，或手工绑定
- 在资产详情页可直接查看该资产的监控指标和告警信息

### 4.2 监控系统

#### 4.2.1 功能概述

监控系统负责采集 IT 基础设施的运行指标数据，提供实时数据展示、历史数据查询和可视化仪表盘能力。监控对象覆盖主机、网络设备和中间件，所有监控数据与 CMDB 资产关联。

#### 4.2.2 功能清单

| 功能模块 | 功能项 | 优先级 | 描述 |
|---------|--------|--------|------|
| 数据采集 | 主机指标采集 | P0 | 通过 Agent 采集 CPU、内存、磁盘、网络等基础指标 |
| 数据采集 | 网络设备采集 | P0 | 通过 SNMP/IPMI 采集交换机、路由器等设备的运行指标 |
| 数据采集 | 中间件采集 | P0 | 通过 Exporter 采集 Nginx、MySQL、Redis、Kafka 等中间件指标 |
| 数据采集 | 自定义指标 | P1 | 支持用户通过 Push Gateway 方式上报自定义业务指标 |
| 数据采集 | 采集配置管理 | P0 | 统一管理采集目标、采集间隔、采集插件配置 |
| 数据存储 | 时序数据存储 | P0 | 监控指标数据存入时序数据库，支持高写入吞吐 |
| 数据存储 | 数据保留策略 | P0 | 支持配置数据保留周期（如：原始数据保留30天，聚合数据保留1年） |
| 数据查询 | PromQL 查询 | P0 | 支持类 PromQL 语法进行指标数据查询和聚合计算 |
| 数据查询 | 指标浏览器 | P1 | 提供可视化的指标选择器，支持按 CMDB 资产筛选指标 |
| 可视化 | 仪表盘管理 | P0 | 支持创建、编辑、分享自定义仪表盘 |
| 可视化 | 预置仪表盘 | P0 | 为每种监控对象预置标准仪表盘（如：Linux 主机概览、MySQL 性能面板） |
| 可视化 | 图表类型 | P0 | 支持折线图、柱状图、仪表盘、热力图、表格等图表类型 |
| 可视化 | 拓扑监控 | P2 | 基于 CMDB 关系拓扑展示各节点实时状态 |

```
              │  · 数据标准化    │
              │  · CMDB 资产关联 │
              │  · 数据预处理    │
              └────────┬────────┘
                       │
              ┌────────┴────────┐
              │  时序数据库      │
              │ VictoriaMetrics │
              └─────────────────┘
```

**采集 Agent 设计要点：**
- 主机 Agent 采用轻量级设计，Go 语言编写，资源占用低（CPU < 1%，内存 < 50MB）
- 支持 Pull（拉取）和 Push（推送）两种采集模式
- Agent 自动向 CMDB 注册自身所属主机信息
- 采集配置通过平台统一下发，支持热更新

### 4.3 告警平台

#### 4.3.1 功能概述

告警平台提供统一的告警规则管理、告警事件处理、通知分发和告警升级能力。告警规则基于监控指标数据触发，告警事件自动关联 CMDB 资产信息，实现从资产维度的告警视图。

#### 4.3.2 功能清单

| 功能模块 | 功能项 | 优先级 | 描述 |
|---------|--------|--------|------|
| 告警规则 | 规则创建 | P0 | 基于监控指标创建告警规则，支持阈值、速率变化等触发条件 |
| 告警规则 | 规则模板 | P1 | 预置常用告警规则模板（如：CPU > 90% 持续5分钟） |
| 告警规则 | 规则分组 | P0 | 按业务线/资产分组管理告警规则 |
| 告警规则 | 规则启停 | P0 | 支持单条规则的启用/禁用 |
| 告警规则 | 静默规则 | P1 | 支持设置维护窗口，在指定时间段内静默特定告警 |
| 告警处理 | 告警收敛 | P0 | 相同告警在指定时间窗口内合并，避免重复通知 |
| 告警处理 | 告警抑制 | P1 | 当高级别告警触发时，自动抑制关联的低级别告警 |
| 告警处理 | 告警认领 | P0 | 运维人员可认领告警，标记处理中状态 |
| 告警处理 | 告警备注 | P1 | 支持对告警事件添加处理备注 |
| 告警处理 | 告警等级 | P0 | 支持 P0(紧急)、P1(重要)、P2(一般)、P3(提醒) 四个等级 |
| 通知分发 | 通知渠道 | P0 | 支持邮件、短信、企业微信/钉钉 Webhook 通知 |
| 通知分发 | 通知策略 | P0 | 按告警等级、资产分组配置不同的通知策略和通知对象 |
| 通知分发 | 通知模板 | P1 | 支持自定义通知消息模板 |
| 告警升级 | 升级策略 | P0 | 告警在指定时间内未处理，自动升级通知上级负责人 |
| 告警升级 | 升级路径 | P1 | 支持多级升级路径配置（如：一线 → 二线 → 经理） |
| 告警视图 | 告警大盘 | P0 | 实时展示当前活跃告警数量、分布、趋势 |
| 告警视图 | 告警列表 | P0 | 告警事件列表，支持按等级、状态、资产、时间等筛选 |
| 告警视图 | 资产告警视图 | P0 | 从 CMDB 资产维度查看关联的所有告警 |
| 告警视图 | 告警统计 | P1 | 告警数量趋势、TOP 告警源、平均处理时长等统计报表 |

#### 4.3.3 告警规则设计

**规则模型：**

```yaml
alert_rule:
  name: "CPU使用率过高"
  description: "服务器CPU使用率超过90%持续5分钟"
  severity: P1                    # 告警等级: P0/P1/P2/P3
  status: enabled                 # enabled / disabled
  asset_group: "production"       # 关联的 CMDB 资产分组
  
  trigger:
    metric: "cpu.usage_percent"   # 监控指标
    operator: ">"                 # 比较运算符: >, <, >=, <=, ==, !=
    threshold: 90                 # 阈值
    duration: "5m"                # 持续时间
    aggregation: "avg"            # 聚合方式: avg, max, min, sum
    
  labels:
    team: "infra"                 # 标签，用于通知路由
    
  annotations:
    summary: "{{ $labels.hostname }} CPU使用率 {{ $value }}%"
    description: "主机 {{ $labels.hostname }} ({{ $labels.ip }}) CPU使用率持续超过90%"
```

**告警等级定义：**

| 等级 | 名称 | 定义 | 响应要求 | 示例 |
|------|------|------|---------|------|
| P0 | 紧急 | 核心业务中断或严重降级 | 5分钟内响应 | 核心数据库宕机、核心网络中断 |
| P1 | 重要 | 部分功能受损或有较大风险 | 15分钟内响应 | CPU/内存持续高位、磁盘空间不足 |
| P2 | 一般 | 非核心功能异常或性能下降 | 30分钟内响应 | 非核心服务异常、性能指标劣化 |
| P3 | 提醒 | 需要关注但暂不影响业务 | 工作时间内处理 | 证书即将过期、配置变更通知 |

#### 4.3.4 告警处理流程

```
指标异常 → 规则匹配 → 触发告警事件
                          │
                    ┌─────┴─────┐
                    │ 告警收敛   │  ← 相同告警去重
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ 告警抑制   │  ← 高级别抑制低级别
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ CMDB 关联  │  ← 补充资产信息
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ 通知分发   │  ← 按策略发送通知
                    └─────┬─────┘
                          │
              ┌───────────┼───────────┐
              │           │           │
         ┌────┴────┐ ┌───┴───┐ ┌────┴────┐
         │ 邮件    │ │ 短信  │ │ Webhook │
         └─────────┘ └───────┘ └─────────┘
                          │
                    ┌─────┴─────┐
                    │ 告警认领   │  ← 运维人员处理
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ 超时未处理? │
                    └─────┬─────┘
                     Yes  │  No
                    ┌─────┴─────┐
                    │ 告警升级   │  ← 通知上级
                    └─────┬─────┘
                          │
                    ┌─────┴─────┐
                    │ 告警关闭   │  ← 指标恢复正常 / 手工关闭
                    └───────────┘
```

#### 4.3.5 告警状态机

```
触发(Firing) → 通知中(Notifying) → 已认领(Acknowledged) → 已恢复(Recovered) → 已关闭(Closed)
                      │                    │
                      │                    └──→ 已关闭(Closed) [手工关闭]
                      │
                      └──→ 升级中(Escalating) → 已认领(Acknowledged)
```

| 状态 | 说明 |
|------|------|
| Firing | 告警条件满足，告警已触发 |
| Notifying | 正在发送通知 |
| Acknowledged | 已被运维人员认领处理 |
| Escalating | 超时未处理，正在升级 |
| Recovered | 指标恢复正常，告警自动恢复 |
| Closed | 告警已关闭（自动恢复或手工关闭） |

### 4.4 工单流程管理

#### 4.4.1 功能概述

工单流程管理用于承接告警处理过程，将告警事件、资产信息、负责人和处理记录串联成可追踪的闭环。平台一期重点支持故障工单，覆盖告警自动建单、工单状态流转、处理记录、闭环确认和告警关闭联动。

#### 4.4.2 功能清单

| 功能模块 | 功能项 | 优先级 | 描述 |
|---------|--------|--------|------|
| 工单生成 | 告警自动建单 | P0 | 告警首次触发后自动创建故障工单，关联告警、资产、等级和负责人 |
| 工单生成 | 重复建单控制 | P0 | 同一活跃告警只创建一张工单，避免重复处理 |
| 工单处理 | 状态流转 | P0 | 支持待处理 → 处理中 → 已完成 → 已闭环的标准流程 |
| 工单处理 | 处理记录 | P0 | 运维人员可追加处理过程、处理结果和关键操作说明 |
| 工单处理 | 权限控制 | P1 | 仅处理人、运维管理员或系统管理员可变更工单状态 |
| 工单流程 | 闭环关闭告警 | P0 | 工单闭环后自动将关联告警状态更新为 Closed |
| 工单视图 | 工单列表 | P0 | 支持按状态、等级、资产、负责人、时间范围筛选和分页 |
| 工单视图 | 工单详情 | P0 | 展示告警信息、资产信息、负责人、处理记录时间线和状态操作 |
| 工单统计 | 基础统计 | P1 | 统计未闭环工单数量、平均处理时长、处理人工作量 |

#### 4.4.3 工单状态机

```
待处理(Pending) → 处理中(Processing) → 已完成(Completed) → 已闭环(Closed)
       │                    │                    │
       └────────────── 可由管理员重新分派或退回 ──────────────┘
```

| 状态 | 说明 | 关键规则 |
|------|------|----------|
| Pending | 工单已创建，等待处理 | 由告警自动生成或管理员手工创建 |
| Processing | 运维人员已开始处理 | 需要记录处理人和开始处理时间 |
| Completed | 处理动作已完成，等待确认 | 需要填写处理结果 |
| Closed | 工单已闭环 | 关闭后不可继续添加处理记录，并联动关闭关联告警 |

#### 4.4.4 告警到工单闭环流程

```
告警触发
  │
  ├── 关联 CMDB 资产和负责人
  │
  ├── 发送通知
  │
  ├── 自动创建故障工单
  │
  ├── 运维人员处理并记录过程
  │
  ├── 标记完成并确认闭环
  │
  └── 自动关闭关联告警
```

**工单核心信息：**

| 信息 | 说明 |
|------|------|
| 关联告警 | 告警 ID、告警名称、等级、触发时间、触发指标和值 |
| 关联资产 | 资产 ID、资产名称、管理 IP、所属分组、负责人 |
| 处理信息 | 处理人、当前状态、处理记录、处理结果、闭环时间 |
| 审计信息 | 创建时间、状态变更时间、操作人 |

#### 4.4.5 用户场景

**场景1：告警自动生成工单**
1. 生产服务器 CPU 使用率持续超过阈值，告警平台生成 P1 告警事件
2. 系统根据资产负责人发送通知
3. 系统自动创建故障工单，状态为待处理，关联告警和资产信息

**场景2：工单处理与告警关闭**
1. 运维人员认领工单并切换到处理中
2. 运维人员记录扩容、清理或配置调整过程
3. 问题解决后将工单标记为已完成并闭环
4. 系统自动关闭关联告警，资产详情页不再显示该活跃告警

## 5. 模块间关联设计

### 5.1 数据流

```
CMDB (资产元数据)
  │
  ├──→ 监控系统：提供资产列表和属性信息
  │     · 采集目标自动从 CMDB 资产生成
  │     · 监控数据标签包含 CMDB 资产 ID
  │
  └──→ 告警平台：提供资产分组和负责人信息
        · 告警规则按 CMDB 资产分组生效
        · 告警通知路由基于 CMDB 资产负责人
  
监控系统 (指标数据)
  │
  └──→ 告警平台：提供实时指标数据
        · 告警规则引擎消费监控指标
        · 告警事件包含触发时的指标值

告警平台 (告警事件)
  │
  ├──→ 工单流程管理：自动创建故障工单并跟踪处理过程
  │
  └──→ CMDB：回写告警状态
        · 资产详情页展示当前告警状态
        · 资产列表展示告警数量统计

工单流程管理 (处理闭环)
  │
  └──→ 告警平台：工单闭环后关闭关联告警
        · 告警列表展示关联工单状态
        · 告警统计纳入处理时长和闭环率
```

### 5.2 关联模型

**核心关联键：** 每个监控对象和告警事件通过 `asset_id` 与 CMDB 资产关联。

```
CMDB Asset (asset_id, hostname, ip, model_type, ...)
     │
     ├── 1:N ──→ Monitor Target (target_id, asset_id, collect_config)
     │                │
     │                └── 1:N ──→ Time Series (metric_name, labels{asset_id}, value, timestamp)
     │                                │
     │                                └── N:1 ──→ Alert Event (event_id, asset_id, metric, value, severity, status)
     │
     └── 1:N ──→ Alert Event (直接关联)
                    │
                    └── 1:1 ──→ Work Order (work_order_id, alert_event_id, assignee_id, status)
```

### 5.3 用户场景示例

**场景1：从资产查看监控和告警**
1. 运维工程师在 CMDB 中查看某台服务器的资产详情
2. 页面下方展示该服务器的实时监控面板（CPU、内存、磁盘等）
3. 同时展示该服务器当前活跃告警和历史告警记录

**场景2：从告警定位到资产**
1. 运维工程师收到 CPU 使用率过高的告警通知
2. 点击告警链接进入告警详情，展示告警关联的资产信息（主机名、IP、所属业务线、负责人）
3. 点击资产链接跳转到 CMDB 资产详情，查看完整资产信息和监控数据

**场景3：资产变更联动监控**
1. IT 管理员在 CMDB 中新增一批服务器资产
2. 系统自动为这些服务器生成监控采集任务
3. 预置的告警规则自动覆盖新资产

**场景4：告警联动工单闭环**
1. 告警事件触发后，系统自动创建关联故障工单
2. 运维人员在工单中记录处理过程并完成闭环
3. 工单闭环后，关联告警自动关闭，告警处理时长计入统计报表

## 9. 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| CMDB | Configuration Management Database | 配置管理数据库，存储所有 IT 资产及其关系信息 |
| CI | Configuration Item | 配置项，CMDB 中管理的每一个资产实体 |
| 模型组 | Model Group | CMDB 中资产的分类组织，如"计算资源"、"网络设备" |
| 模型 | Model | 模型组下的具体资产类型定义，包含字段和关系 |
| 指标 | Metric | 监控系统中采集的数值型时间序列数据 |
| 采集器 | Collector/Exporter | 负责从监控目标收集指标数据的组件 |
| 告警规则 | Alert Rule | 定义告警触发条件的规则，包含指标、阈值、持续时间等 |
| 告警事件 | Alert Event | 告警规则触发后生成的具体告警实例 |
| 告警收敛 | Alert Deduplication | 将相同告警在时间窗口内合并，避免重复通知 |
| 告警抑制 | Alert Suppression | 高级别告警触发时自动抑制关联的低级别告警 |
| 告警升级 | Alert Escalation | 告警超时未处理时自动通知更高级别负责人 |
| 工单 | Work Order | 承接告警或运维事项的处理记录和闭环流程 |
| 仪表盘 | Dashboard | 监控数据的可视化面板，由多个图表组件组成 |
| 时序数据库 | Time Series Database | 专门用于存储时间序列数据的数据库 |
| SLA | Service Level Agreement | 服务等级协议，定义服务可用性承诺 |
| RBAC | Role-Based Access Control | 基于角色的访问控制 |
